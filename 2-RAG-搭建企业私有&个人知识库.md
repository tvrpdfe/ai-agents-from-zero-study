**本章课程目标：**

- 理解 RAG 是什么、为什么需要、以及「检索 → 增强 → 生成」的完整执行流程（含知识库构建与查询两阶段）。

## 1、RAG 的理解

### 1.1 什么是 RAG

RAG（Retrieval-Augmented Generation，检索增强生成）是一种结合`信息检索`（Retrieval）与`文本生成`（Generation）的技术，旨在提升大语言模型在回答专业问题时的`准确性`和`可靠性`。

> **可这样记：** RAG = **先查你的资料（检索），再把查到的内容塞进问题里（增强），最后让大模型根据这些资料生成答案（生成）**。大模型「不知道」或「容易说错」的那部分，用你自己的文档来补，所以适合企业制度、产品手册、个人笔记、最新资料等场景。与 [1-3](1-3-RAG、微调、续训与智能体选型.md) 中的「何时需要 RAG」一致：缺知识用 RAG，缺行为才考虑微调。

### 1.2 为什么需要 RAG

- 知识更新：解决大模型静态知识无法实时更新的问题
- 领域适配：通过私有知识库实现垂直领域知识增强
- 成本控制：相比微调方案具有更低的实施成本

**背景：**

- 大模型的知识冻结
- 大模型幻觉

**举例 1：**

随着 LLM 规模扩大，训练成本与周期相应增加。因此，包含最新信息的数据难以融入模型训练过程，无法及时反映最新的信息或动态变化，导致 LLM 难以应对诸如“请推荐当前热门影片”等时间敏感性问题。

**举例 2：**

大型语言模型（LLM）的训练依赖于网络上`海量公开的静态数据`，而某些`特定领域`（如企业内部资料、专有技术文档等）的数据通常不会作为公开的训练数据，导致模型在面对这些领域的查询时，可能因缺乏足够的信息而生成不准确甚至虚构的回复。

**解决方案：**

为了解决这一问题，RAG 技术通过引入`向量数据库（Vector Database）`作为外部知识源，将模型缺失的知识以结构化的形式提供。

**举例 1：**

LLM 在考试的时候面对陌生的领域，答复能力有限，然后就准备放飞自我了，而此时 RAG 给了一些提示和思路，让 LLM 懂了开始往这个提示的方向做，最终考试的正确率从 60%到了 90%！

### 1.3 执行流程

![langchain+chatglm](images/2/2-1-3-1.png)

**流程说明：**

整张图分为**两个阶段**：上方是**知识库构建**，下方是**查询与答案生成**。结合前文「为什么需要 RAG」——解决大模型知识冻结、幻觉、无法访问私有/最新数据——本图展示的正是：如何用 **LangChain + ChatGLM** 把本地文档变成可检索的知识库，再用检索到的内容增强大模型回答。

| 步骤                       | 英文名              | 中文名         | 在做什么                                                                                                                                                                 |
| -------------------------- | ------------------- | -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **阶段一：知识库构建**     |
| 1                          | Local Documents     | 本地文档       | 流程起点：你的 PDF、Word、Markdown、TXT 等原始文件，是私有知识库的数据来源。                                                                                             |
| 2                          | Unstructured Loader | 非结构化加载器 | 读取并解析各种格式的本地文档，从中抽出纯文本，统一成后续可处理的格式（对应 LangChain 的 DocumentLoader）。                                                               |
| 3                          | Text                | 文本           | 加载器产出的纯文本内容。                                                                                                                                                 |
| 4                          | Text Splitter       | 文本分割器     | 把长文本按规则（字符数、段落、递归等）切成小块，便于嵌入和放入 LLM 上下文，对应多种 TextSplitter。                                                                       |
| 5                          | Text Chunks         | 文本块         | 分割后得到的、长度合适的一段段文本，是后续「嵌入」和「检索」的基本单位。                                                                                                 |
| 6                          | Embedding           | 嵌入           | 用嵌入模型把每个文本块转成高维向量，向量能表示语义；语义相近的块，向量距离更近。                                                                                         |
| 7                          | VectorStore         | 向量存储       | 把文本块的向量和原文存进向量数据库（如 Chroma、FAISS、Pinecone），支持按相似度快速检索。                                                                                 |
| **阶段二：查询与答案生成** |
| 8                          | Query               | 查询           | 用户用自然语言提出的问题或请求。                                                                                                                                         |
| 9                          | Embedding           | 嵌入           | 用**同一套**嵌入模型把用户查询也转成向量，保证和知识库在同一向量空间，才能做相似度比较。                                                                                 |
| 10                         | Query Vector        | 查询向量       | 用户查询的向量表示。**对应 RAG 里的「检索」环节**：用该向量去向量库里找最相关的块。                                                                                      |
| 11                         | Vector Similarity   | 向量相似度     | 在向量存储中做相似度搜索，找出与查询向量最接近的文本块。                                                                                                                 |
| 12                         | Related Text Chunks | 相关文本块     | 检索得到的、与问题最相关的文本片段，是回答问题的依据和上下文。                                                                                                           |
| 13                         | Prompt Template     | 提示模板       | 预定义模板，把「用户查询」和「相关文本块」拼成结构化提示（如「根据以下信息回答：[相关文本块] 问题：[查询]」）。**对应 RAG 里的「增强」**：用检索结果增强发给模型的提示。 |
| 14                         | Prompt              | 提示           | 最终送入大模型的那段话：查询 + 检索到的上下文。                                                                                                                          |
| 15                         | LLM                 | 大语言模型     | LangChain + ChatGLM：根据提示中的上下文和问题生成回答。**对应 RAG 里的「生成」**。                                                                                       |
| 16                         | Answer              | 答案           | 模型基于检索到的私有知识生成的最终自然语言答案。                                                                                                                         |

![RAG架构图](images/2/2-1-3-2.svg)

> 检索-增强-生成过程：**检索**对应第 9 ～ 11 步（查询嵌入 → 查询向量 → 向量相似度搜索）；**增强**对应第 13 步（提示词中注入检索到的数据）；**生成**对应第 15 步（LLM 输出答案）。

> **初学者常问：** 为什么「建知识库」和「用户提问时」都要用**同一套**嵌入模型？因为嵌入模型把文本变成向量，语义相近的文本向量才接近；建库用模型 A、检索用模型 B，两边的向量不在同一套「尺子」下，相似度比较会乱，检索就失灵。所以平台里配置的「嵌入模型」建库和检索是同一份，不要混用不同模型。

## 2、知识库的概述

### 2.1 哪些人需要搭建（个人）知识库

**小型企业主或创业者：**查阅和分享文件、文档、客户反馈、市场分析，大大提升你的工作效率。

**职场打工人或自由职业者：**无论是写作、设计、开发，还是视频制作，知识库都可以管理大量的素材、创意和客户需求，通过知识库，你可以轻松存储和搜索这些资料，并通过大模型进行二次创作。

**教育工作者或学生：**利用知识库管理教学资源、课程安排、教材资料等，学生则可以将课堂笔记、参考书目和作业整理在一起，随时复习和备考。

**生活中的普通人：**无论是旅行计划、兴趣爱好，还是学习笔记，全部都可以集中在知识库管理。

### 2.2 知识库各个搭建平台对比

很多平台都支持个人知识库的搭建。

#### 2.2.1 核心定位和技术特点

**AnythingLLM、CherryStudio**：`桌面/图形化 AI 助手 + 知识库（RAG）`，支持对接云模型与本地模型；适合`个人/小团队`快速验证。在“多租户治理、复杂系统集成、生产化观测”等方面通常不如平台型/专业引擎。

**Dify、FastGPT**：`LLM Agent 与工作流编排平台`，支持创建知识库，支持`云端`与`本地`部署（可用 Docker）。

**RAGFlow**：基于`深度文档理解`的开源 RAG 引擎，强调复杂文档解析、引用与可视化干预；也支持 Agent 能力，但通用工作流生态通常不如“全能型 Agent 平台”。（支持 Docker 部署）

#### 2.2.2 典型场景与选型建议

**1. 个人知识管理（轻量级）**

- 需求：快速验证、低预算（开发成本 <= 1 周），个人/小团队使用，**以“能用”为主**（文档以 Markdown、PDF、网页为主）。

- 推荐工具：Cherry Studio / AnythingLLM

- 理由：
  - 部署和操作简单，上手快
  - 可直接对接在线大模型 API，也可接本地模型，适合快速试错
  - 支持多模型对话（如 DeepSeek + Ollama），适合整合笔记/文献

**2. 应用化交付与团队协作（平台型场景）**

- 需求：
  - 将知识库能力**封装为可复用的 AI 应用或 Agent**
  - 支持多成员协作、权限控制、应用发布与版本迭代
  - 需要流程编排（如检索 → 工具调用 → 多轮推理），而不仅是简单问答
- 推荐工具：Dify / FastGPT
- 理由：
  - 提供完整的 **Agent / Workflow 编排能力**，知识库（RAG）作为其中一环
  - 支持多应用管理、角色与权限控制，更适合团队或内部平台使用
  - 易于与业务系统集成（API / Webhook），便于“从 Demo 走向可用系统”

**3. 企业级文档解析（高精度需求）**

- 需求：面向复杂文档（**长 PDF、复杂版式、表格/图片混排、扫描件**等），强调解析质量与可控性，要求可追溯引用。
- 推荐工具：RAGFlow
- 理由：
  - 强调深度文档理解，解析/分块结果可视化，便于检查与必要时干预
  - 对复杂格式更友好，适合把“文档解析质量”作为核心竞争力的场景

## 5、使用 Dify 搭建知识库

### 5.1 Dify 介绍

Dify（DefineModify）是一个开源的大语言模型(LLM)应用开发平台，由`苏州语灵人工智能`推出。

Dify 为 AI Agent 提供了 50 多种内置工具，如谷歌搜索、DALL·E、Stable Diffusion 和 WolframAlpha 等。

官网：https://dify.ai/zh

说明：https://github.com/langgenius/dify/blob/main/README_CN.md

### 5.2 源数据格式

通过使用 Dify，可以方便快捷地构建私有知识库。可以将知识库放在工作流中，协同多种工具一起使用。而且 Dify 提供的知识库功能有着简洁的可视化界面，可以很方便地进行管理，适用于个人和团队。

目前 Dify 支持多种源数据格式，包括：

- 长文本内容：TXT、Markdown、DOCX、HTML、JSON、PDF

- 结构化数据：CSV、Excel

**注：** 私有知识库要达到良好效果，需配合 **Embedding 模型**与 **Rerank 模型**使用。请在 Dify 的模型供应商配置（或所对接的推理服务，如 Xinference）中启用相关模型并接入 Dify。

### 5.3 构建私有知识库

**步骤 1：首先创建一个新的知识库**

![Dify创建知识库](images/2/2-5-3-1.png)

**步骤 2：上传知识库文件**

这里准备的是一份《刑法》的 TXT 格式文本，按自然段划分了每一条法条。

![Dify上传文件](images/2/2-5-3-2.png)

**步骤 3：分段设置**

大语言模型存在有限的上下文窗口，通常需要将整段文本进行分段处理后，将与用户问题关联度最高的几个段落召回，即分段 top-K 召回模式。此外，在用户问题与文本分段进行语义匹配时，合适的分段大小将有助于匹配关联性最高的文本内容，减少信息噪音。

分段标识符如果是`\n`，则是以换行为一个分段；如果是`\n\n`，则是以一个段落为一个分段。点击`预览块`查看目前块划分的情况。

分段重叠长度一般是分段最大长度的`10%-20%`。

知识库文档中若有 URL、邮箱等，可在预处理时过滤掉。

**分段设置说明（这些项是干嘛的、对知识库有什么影响）**

| 配置项           | 作用                                                                                                                   | 对知识库的影响                                                                                                                                                                                                                                           |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **分段标识符**   | 决定「按什么边界」把长文切成小块。`\n` = 按**换行**切，一块往往是一行或几行；`\n\n` = 按**空行**切，一块往往是一整段。 | 选得合适，切出来的块语义完整，检索时更容易命中「一整段有用信息」；选得不合适（例如长文只用 `\n`），可能把一段话拆碎，检索到的块缺上下文。                                                                                                                |
| **分段最大长度** | 单块文本的**字数/字符上限**，超过就再切一刀。                                                                          | **设太大**：一块里塞进很多内容，和用户问题「相似」的只是一小句，整块都被召回，容易带进无关内容，答案噪音多。**设太小**：一句完整的话被拆成多块，语义被拆散，检索容易漏掉或只命中半句。一般建议 200 ～ 800 字（或按 token 约 100 ～ 500）视文档类型微调。 |
| **分段重叠长度** | 相邻两块之间**重复一段文字**的长度，避免一刀切在句子中间导致语义断裂。                                                 | 有一定重叠（如最大长度的 10%～ 20%），边界上的句子不会因为被切断而「半句在这一块、半句在下一块」，检索到任一块都能拿到完整一句，上下文更完整。重叠太大则重复多、浪费存储和算力。                                                                         |

![Dify分段设置](images/2/2-5-3-3.png)

**步骤 4：选择索引方式**

这里自动选择高质量。高质量的准确性更高，但是 token 消耗也会增加。如果使用的是部署到本地的模型，花费就没有影响。

![Dify索引方式](images/2/2-5-3-4.png)

还有 **Q&A 方式**：若文档本身为问答对形式，则选择此方式最契合。

**步骤 5：检索设置**

在此可选择 **Embedding 模型**与 **Rerank 模型**，也可以设置 Top K，也就是选出最相似的前 n 条。选择 Score 阈值，即筛选文本的相似度阈值。

![Dify检索设置](images/2/2-5-3-5.png)

混合检索：既包括向量检索（可选用 Rerank 模型做精排），也包含全文检索。

**检索设置说明（这些项是干嘛的、对知识库有什么影响）**

| 配置项             | 作用                                                                                                                    | 对知识库的影响                                                                                                                                                                                                                   |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Embedding 模型** | 把「文档块」和「用户问题」都变成向量，用来算**语义相似度**，从而做检索。                                                | 建库和检索必须用**同一个** Embedding 模型，否则向量对不上、检索乱掉。选主流模型（如 BGE、E5、GTE）一般效果更稳。                                                                                                                 |
| **向量检索**       | 用问题向量在向量库里找「最像」的文档块，按相似度排序。                                                                  | 是 RAG 的「找相关段落」核心步骤；没有向量检索，就无法按语义从知识库里捞内容。                                                                                                                                                    |
| **Rerank 模型**    | 在向量检索拿到一批候选块之后，再用一个专门模型对「问题 + 每条候选」打**相关性分**，按分数重新排序，只保留最相关的几条。 | **不用 Rerank**：只靠向量相似度，有时会混进意思略相关但答非所问的块。**用 Rerank**：精排后交给大模型的内容更准，答案质量更好，但多一步调用、略增延迟和成本。文档多、要求高时建议开。                                             |
| **Top K**          | 检索时**最多取几条**文档块交给后续（或交给 Rerank）。例如 Top K = 5 表示只保留相似度最高的 5 条。                       | **K 太小**（如 1 ～ 2）：可能漏掉其他相关段落，答案信息不全。**K 太大**（如 20 ～ 30）：无关块多，上下文噪音大，模型容易跑题或重复，且 token 消耗大。一般 5 ～ 10 条常用，可先试 5，再按效果调。                                 |
| **Score 阈值**     | 相似度**下限**：只有相似度 ≥ 这个分数的块才会被保留，低于的直接过滤掉。                                                 | **阈值设高**：只保留非常相关的块，结果少、干净，但可能把「有点相关」的有用内容也滤掉，答案略单薄。**阈值设低**：保留更多块，信息多但噪音也可能多。可根据「预览块」或测试问答效果微调，找不到内容就适当降阈值，噪音多就适当提高。 |
| **混合检索**       | 同时做**向量检索**（按语义）和**全文检索**（按关键词匹配），再把两种结果合并、去重、排序。                              | 用户问题里若有关键词（如法条编号、专有名词），全文检索能兜住；向量检索负责语义。两者一起用，既照顾「原词出现」又照顾「意思相关」，适合文档既有自然段又有条目标号的场景（如法律、手册）。                                         |

设置完成后，保存并处理即可。

![Dify保存处理](images/2/2-5-3-6.png)

**章节思考题：**

1. 为什么说 RAG 不是训练模型，而是给模型临时补上下文？

   **答案：** 因为 RAG 不会修改模型参数，它做的是在提问时先检索相关文档，再把这些文档拼进上下文里让模型基于资料回答。所以它更像“临时开卷答题”，而不是“重新训练大脑”。

2. 知识库构建阶段和查询阶段分别在做什么？

   **答案：** 知识库构建阶段主要做文档加载、清洗、切块、向量化和入库；查询阶段主要做问题向量化、相似检索、必要的重排，再把命中的内容交给模型生成答案。前者是在准备资料，后者是在按题找资料并作答。

3. Top K、阈值、Rerank 这几个参数为什么会直接影响最终回答质量？

   **答案：** 因为它们决定了召回多少内容、保留哪些结果、是否让更相关的片段排到前面。参数过大容易把噪声塞进上下文，参数过小又可能漏掉关键信息，所以它们会直接影响最终答案的相关性和准确率。

4. 如果你要给一个刚入门的同学解释“知识库问答为什么比直接问模型更靠谱”，你会怎么讲？

   **答案：** 可以这样讲：直接问模型，它只能靠训练时学过的知识回答，遇到公司内部资料常常会不知道或乱猜；知识库问答会先去你的文档里找证据，再让模型基于证据回答，所以更贴近企业真实资料，也更容易追溯答案来源。

5. 回想你自己的学习或工作资料，试着判断哪一类文档最适合先拿来做一个小型 RAG 知识库。

   **答案：** 最适合先做小型 RAG 的，通常是结构相对清晰、价值高、更新频率适中、经常被反复查询的资料，比如产品手册、内部 FAQ、制度文档、项目知识沉淀。它们既能快速体现效果，也便于验证切块和召回质量。

**本章小结：**

- **RAG 是什么**：检索（从你的文档里找相关段落）→ 增强（把段落塞进提示）→ 生成（大模型基于这些内容回答）；解决知识冻结、幻觉、私有/最新数据不可用等问题。
- **执行流程**：两阶段——（1）知识库构建：本地文档 → 加载 → 分块 → 嵌入 → 存入向量库；（2）查询与生成：用户问题 → 嵌入成向量 → 向量相似度检索 → 相关块 + 问题拼成提示 → LLM 生成答案。建库和检索必须用同一套嵌入模型。
- **谁需要知识库**：个人（笔记、写作、学习）、小团队（文档共享、客户反馈）、教育、企业（制度、手册、合同等）；按「个人轻量 / 平台化协作 / 复杂文档解析」选 Cherry Studio·ima、Dify·FastGPT、RAGFlow 等。
- **三个动手平台**：**Cherry Studio**、**ima** 适合个人快速上手，配置 API 与嵌入模型即可建库、检索、问答；**Dify** 适合要和工作流、Agent 结合的团队，支持分段与检索参数（分段标识符、最大长度、重叠、Embedding/Rerank、Top K、阈值、混合检索）的细调，效果更好、可追溯引用。
- **效果相关**：分段大小、重叠、嵌入模型、是否启用 Rerank、Top K 与阈值都会影响「查得准不准、答得稳不稳」；手写/表格/复杂公式多的文档可先用 Doc2X 等做解析再上传。
- 从掌握结果看，学完本章后，你至少应该：能说清 RAG 的两阶段流程：知识库构建阶段、查询与生成阶段；能建立“文档 → 分块 → 嵌入 → 向量库 → 检索 → 拼进提示 → 生成答案”的直觉；能根据个人 / 团队 / 工作流需求，大致判断 Cherry Studio、ima、Dify 等平台的适用场景。
